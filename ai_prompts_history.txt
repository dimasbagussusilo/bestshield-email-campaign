## The worker pool pattern in Go

The canonical bounded worker pool:

```go
func main() {
    numWorkers := 1000
    jobs := make(chan Job, jobBufferSize)
    results := make(chan Result, resultBufferSize)

    // Start workers
    var wg sync.WaitGroup
    for w := 0; w < numWorkers; w++ {
        wg.Add(1)
        go worker(jobs, results, &wg)
    }

    // Producer
    go func() {
        for j := 0; j < numJobs; j++ {
            jobs <- generateJob(j)
        }
        close(jobs)  // signal no more work
    }()

    // Closer: wait for workers, then close results
    go func() {
        wg.Wait()
        close(results)
    }()

    // Collector: drain results
    for r := range results {
        aggregate(r)
    }
}
```

# Spec: High-throughput promotional email sender (dry run)

## Context

Goal: deliver a personalized promotional email to **1,000,000** customer addresses *as fast as
reasonably possible*, as a **dry run** — no real email is sent. Each "send" is a simulated network
call that sleeps **10–50ms**. The exercise is really about **safe, bounded concurrency in Go**: we
must not spawn 1,000,000 goroutines (≈2–8GB of stacks) or hold 1,000,000 jobs in memory at once.

Chosen approach: a fixed-size **worker pool** fed by channels. This bounds goroutines and memory
while letting throughput scale with the pool size. Decisions confirmed with the user:
- **Default worker pool = 1,000** (safe/idiomatic; ~33k jobs/sec → ~30s for 1M; tunable).
- **Input = synthetic generation** (producer fabricates records on the fly; zero disk I/O).

## Architecture (single `main.go`, module `tamakun`, Go 1.22+)

```
producer goroutine ──jobs chan──▶ N worker goroutines ──results chan──▶ collector (main)
   (generate 1M        (buffered)    (sendEmail: sleep      (buffered)      (aggregate +
    Jobs, stream)                     10-50ms, emit Result)                 drain until closed)
```

Lifecycle is driven entirely by `close()`:
1. Producer streams Jobs until done (or ctx cancelled) → `close(jobs)`.
2. Workers `for job := range jobs` → exit when `jobs` closed → `wg.Done()`.
3. A closer goroutine `wg.Wait()`s on all workers → `close(results)`.
4. Collector (main) `for r := range results` until `results` closed, then prints stats.

This is the canonical, race-free pattern: bounded goroutines, bounded memory, no busy-waiting.

## Tunables (flags; defaults shown)

| Flag | Default | Note |
|---|---|---|
| `-total` | `1000000` | number of recipients |
| `-workers` | `1000` | pool size — the central knob |
| `-min-ms` / `-max-ms` | `10` / `50` | simulated latency range |
| `-job-buffer` / `-result-buffer` | `= workers` | channel capacities |
| `-fail-rate` | `0` | optional synthetic failure % to exercise the error path |

## Channel buffer sizes

```go
jobs    := make(chan Job,    workers) // 1,000
results := make(chan Result, workers) // 1,000
```
- **`jobs` = `workers`**: total in-flight ≈ `2 × workers`. Tiny memory (1k Jobs × ~64B ≈ 64KB),
  gives each worker one queued job ahead to cover scheduler jitter, and applies clean backpressure
  to the producer (it blocks once workers are saturated — which is *correct* and memory-safe).
- **`results` = `workers`**: the collector drains far faster than workers produce (counter increments
  ≈ ns vs 30ms/result), so this never fills; sized to workers so a worker never blocks on send even
  if the collector lags briefly. Both are flags if we want to experiment.

```go
    results := make(chan Result, workers)
```

40	## Architectural decisions
41	
42	### Why a bounded pool of 1,000 workers instead of 1,000,000 goroutines?
43	
44	The naive approach — one goroutine per recipient — would spawn **1M goroutines**,
45	each with a 2–8KB stack: roughly **2–8 GB of memory**, plus heavy scheduler and
46	GC churn. It is unbounded and resource-hostile.
47	
48	Instead we use a **fixed pool of workers** connected by buffered channels:
49	
50	```
51	producer ──jobs(chan, 1000)──▶ 1000 workers ──results(chan, 1000)──▶ collector
52	```
53	
54	- **Bounded resources:** goroutines are capped at the pool size. In every test,
55	  `runtime.NumGoroutine()` peaked at **1,005** (1000 workers + ~5 overhead) —
56	  never 1,000,000.
57	- **Bounded memory:** only `~2 × workers` jobs are ever in flight (~128 KB of
58	  Job/Result structs), and channels provide natural **backpressure** — the
59	  producer simply blocks when workers are saturated.
60	- **Throughput scales with the knob:** since simulated work is pure I/O-style
61	  sleep, throughput ≈ `workers / meanLatency`. At a 30ms mean: 1,000 workers ≈
62	  **32k/s (~30s)**; 5,000 ≈ **163k/s (~6s)**. 1,000 is the safe, idiomatic
63	  default; it is a flag, not a constant.
64	
65	Completion is driven entirely by `close()` — the producer closes `jobs`, workers
66	exit their range loop, a closer goroutine waits on the workers then closes
67	`results`, and the collector drains until closed. This makes the pipeline
68	**race-free** without any mutexes or atomics: the single collector goroutine is
69	the sole owner of the aggregated stats.
70	
71	**Other choices:** `math/rand/v2` for lock-free latency RNG across workers
72	(no global mutex); `time.NewTimer` + `Stop()` instead of `time.After` to avoid
73	leaving a timer behind on each of 1M calls; and `signal.NotifyContext` for
74	graceful Ctrl-C cancellation.
75	
76	### Why 1,000 (and not more)?
77	
78	For the dry run you can safely raise `-workers` since the work is pure sleep.
79	In a **real** deployment, however, the limit is the downstream SMTP server's
80	allowed concurrency — not this knob — so over-tuning the pool size is beside the
81	point. 1,000 is a reasonable, honest default; the flag lets the reviewer explore
82	the speed/resource tradeoff directly.
